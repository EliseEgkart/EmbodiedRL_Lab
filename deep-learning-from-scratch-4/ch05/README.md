# Chapter 5: Monte Carlo Methods

## Core Idea

Monte Carlo methods estimate values from complete sampled returns rather than exact expectations. If dynamic programming uses the model to average over all possible futures, Monte Carlo waits for an actual trajectory and averages realized outcomes.

## Return

For an episode starting at time `t`,

`G_t = R_{t+1} + gamma R_{t+2} + gamma^2 R_{t+3} + ... + gamma^{T-t-1} R_T`

Monte Carlo methods use `G_t` directly as a target.

## Monte Carlo Policy Evaluation

For a fixed policy `pi`,

`V^pi(s) = E_pi[G_t | S_t = s]`

An empirical estimator is

`V(s) <- average of observed returns following visits to s`

[mc_eval.py](/Users/HyeongjinKim/OneDrive/Desktop/EmbodiedRL_Lab/deep-learning-from-scratch-4/ch05/mc_eval.py) computes this by storing a full episode in memory and then traversing it backward to build returns recursively.

## First-Visit and Every-Visit View

There are two standard variants:

- first-visit MC: use only the first time a state appears in an episode,
- every-visit MC: use all visits.

The code here effectively averages episode-derived returns over visits without explicitly separating those cases.

## Monte Carlo Control

To learn action values instead of state values, use

`Q(s, a) <- average of returns observed after taking action a in state s`

Then improve the policy greedily or epsilon-greedily with respect to `Q`.

[mc_control.py](/Users/HyeongjinKim/OneDrive/Desktop/EmbodiedRL_Lab/deep-learning-from-scratch-4/ch05/mc_control.py) uses a constant-step update

`Q <- Q + alpha * (G - Q)`

instead of exact counting. This is often preferable in code because it is simpler and extends naturally to non-stationary or approximate settings.

## Importance Sampling

Off-policy Monte Carlo asks a harder question:

- data is generated by a behavior policy `b`,
- but we want to evaluate or improve a different target policy `pi`.

To correct for this mismatch, we multiply returns by the likelihood ratio

`rho_{t:T-1} = product_{k=t}^{T-1} pi(A_k | S_k) / b(A_k | S_k)`

Then

`E_b[rho G_t] = E_pi[G_t]`

under standard support assumptions.

[importance_sampling.py](/Users/HyeongjinKim/OneDrive/Desktop/EmbodiedRL_Lab/deep-learning-from-scratch-4/ch05/importance_sampling.py) demonstrates the same idea in a tiny discrete expectation example before applying it to control in [mc_control_offpolicy.py](/Users/HyeongjinKim/OneDrive/Desktop/EmbodiedRL_Lab/deep-learning-from-scratch-4/ch05/mc_control_offpolicy.py).

## Bias-Variance Tradeoff

Monte Carlo targets are unbiased because they use actual full returns, but they often have high variance:

- long episodes accumulate noise,
- stochastic transitions and rewards amplify target spread,
- importance weights can explode in off-policy learning.

This high variance is the main reason later chapters introduce temporal-difference bootstrapping and actor-critic baselines.

## Why the Episode Buffer Exists

The `memory` list in the code is mathematically necessary. Monte Carlo cannot update from one transition alone because the target for time `t` depends on all rewards until termination.

That is why the code structure is:

1. collect an entire episode,
2. traverse backward,
3. accumulate `G`,
4. update value estimates.

## Implementation Insights

- Backward traversal is the cleanest way to compute returns because `G_t = R_{t+1} + gamma G_{t+1}`.
- Epsilon-greedy improvement in Monte Carlo control is already a generalized policy-iteration loop, but with sampled evaluation instead of exact DP evaluation.
- Constant-step updates trade exact averaging for simpler online-style code and better adaptability.
- Off-policy Monte Carlo is mathematically elegant but can be numerically unstable because importance ratios may grow rapidly.

## Connection to TD

Monte Carlo uses the full target `G_t`. TD will replace that expensive target by the one-step approximation

`R_{t+1} + gamma V(S_{t+1})`

which introduces bias but often reduces variance dramatically.
